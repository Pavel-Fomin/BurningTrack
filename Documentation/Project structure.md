# Назначение

Project Structure описывает организацию проекта BurningTrack и правила размещения кода по зонам ответственности.

Документ отвечает только на вопрос: где должна располагаться каждая категория кода. Он не объясняет архитектуру приложения, обязанности root screens, navigation-flow или отдельных feature-подсистем; эти темы раскрываются в специализированных архитектурных документах.

# Проблема

Если файлы размещаются только по текущему удобству, проект быстро теряет устойчивую структуру: одинаковые сущности оказываются в разных местах, feature-код смешивается с общими компонентами, а инфраструктурные зависимости начинают проникать в UI-слой.

Хаотичное расположение файлов усложняет поиск владельца кода, повышает риск дублирования и делает внутренние рефакторинги дороже. Поэтому структура проекта должна отражать назначение кода и его архитектурную ответственность.

# Принципы организации

- Проект разделяется по ответственности.
- Каталог должен показывать назначение кода, а не только его технический тип.
- Новая функциональность размещается рядом со своей подсистемой и не выносится в общую область без необходимости.
- Общие компоненты не размещаются внутри feature-подсистем.
- Feature-specific код не переносится в Shared только ради удобного доступа.
- Инфраструктура и внешние интеграции отделяются от UI.
- Документация хранится отдельно от кода и организуется по назначению документа.

# Верхний уровень структуры проекта

Конкретные названия каталогов могут изменяться. Документ описывает архитектурные области проекта, а не точную файловую структуру.

- Код приложения — объединяет остальные архитектурные области в единое приложение.
- UI — отвечает за отображение пользовательского интерфейса.
- Domain — отвечает за предметную область приложения.
- Infrastructure — отвечает за взаимодействие приложения с внешним миром и системными возможностями.
- Shared — содержит общие архитектурные элементы, используемые несколькими подсистемами.
- Resources — содержит ресурсы приложения.
- Documentation — содержит документацию проекта.
- External Libraries — содержит внешние зависимости проекта.
- Tests — содержит код проверки приложения.

# Правила размещения кода

Новая feature-подсистема размещается по архитектурной ответственности: пользовательский интерфейс — в UI, предметные правила — в Domain, взаимодействие с внешним миром и системными возможностями — в Infrastructure, общие элементы — в Shared.

UI размещается в UI-области проекта. UI не становится местом для Domain-правил, инфраструктурных зависимостей или внешних интеграций.

Domain размещается в области Domain. Код этой области описывает предметную область приложения и не зависит от способа отображения пользовательского интерфейса.

Infrastructure размещается в инфраструктурной области. Код этой области отвечает за взаимодействие приложения с внешним миром, системными возможностями и внешними зависимостями.

Shared используется только для общих архитектурных элементов, которые нужны нескольким подсистемам. Feature-specific код остаётся в области своей подсистемы, пока не появляется устойчивая общая ответственность.

Resources содержит ресурсы приложения и не становится местом для логики приложения или архитектурных контрактов.

External Libraries содержит внешние зависимости проекта. Код подключения внешней зависимости относится к Infrastructure, если он описывает взаимодействие приложения с этой зависимостью.

Tests содержит код проверки приложения. Проверочный код размещается отдельно от проверяемых архитектурных областей и не меняет их ответственность.

Documentation содержит документацию проекта. Документы разделяются по назначению и ссылаются на каноничные архитектурные документы, если вопрос уже раскрыт.

# Контракты

- Каждый новый файл размещается по владельцу ответственности.
- Feature-код остаётся внутри своей feature-области, пока не появляется устойчивая общая ответственность.
- Shared-код используется только для действительно общих компонентов, контрактов и поддерживающей функциональности.
- Shared-область не зависит от конкретной feature-подсистемы.
- UI-слой не становится владельцем domain-логики или инфраструктуры.
- Инфраструктурный код не размещается внутри root screen или feature UI только потому, что используется экраном.
- Внешняя библиотека не используется напрямую из разных частей приложения без слоя подключения.
- Документация не дублирует архитектурные документы и ссылается на каноничный документ, если вопрос уже раскрыт.

# Инварианты

- Одна ответственность имеет одно место в структуре проекта.
- Одинаковые категории сущностей располагаются одинаково.
- Feature не становится Shared.
- Shared не зависит от Feature.
- Инфраструктура отделена от UI.
- Domain-логика не размещается в presentation-слое.
- Внешние интеграции изолированы от feature UI.
- Структура проекта отражает архитектуру, но не заменяет архитектурную документацию.
