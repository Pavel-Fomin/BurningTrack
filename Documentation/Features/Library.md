# Назначение

Library — feature-подсистема фонотеки BurningTrack.

Она решает пользовательскую задачу доступа к музыкальной коллекции: пользователь подключает музыкальные папки, просматривает структуру фонотеки, открывает папки с треками и запускает действия над отдельными или выбранными треками.

Library не определяет архитектурные правила других подсистем. Она использует соответствующие архитектурные контракты и остаётся feature-подсистемой пользовательской фонотеки.

# Пользовательский сценарий

Пользователь открывает Library, чтобы работать со своей музыкальной коллекцией.

Если фонотека ещё не подключена, пользователь выбирает музыкальную папку. После подключения Library показывает доступную структуру фонотеки и позволяет переходить по папкам.

Внутри папки пользователь просматривает треки, запускает воспроизведение, открывает действия над отдельным треком, добавляет треки в пользовательские треклисты или выбирает несколько треков для массового действия.

Если действие требует временного пользовательского сценария, Library использует соответствующий presentation-контракт. Если действие завершается успешно, частично успешно или с ошибкой, Library передаёт результат в контракт пользовательской обратной связи.

# Основной поток

Типовой поток Library:

```text
Пользователь открывает Library
↓
Library показывает состояние фонотеки
↓
Пользователь подключает папку или открывает существующую
↓
Library показывает структуру папок или список треков
↓
Пользователь выбирает действие над треком или группой треков
↓
Library передаёт намерение в соответствующую подсистему
↓
Результат возвращается через соответствующий архитектурный контракт
↓
Library обновляет отображаемое состояние
```

Поток не фиксирует внутреннее устройство используемых архитектурных контрактов. Эти вопросы раскрываются в каноничных архитектурных документах.

# Участвующие подсистемы

- Root Screens — Library подключается как самостоятельная feature-подсистема через Root Screens и использует общий screen-flow контракт.
- Navigation — Library использует Navigation для переходов внутри фонотеки и переходов в Library из других сценариев.
- Runtime Metadata — Library использует Runtime Metadata для актуальных данных треков в отображении и действиях.
- Track Identity — Library использует Track Identity для действий, относящихся к физическому треку.
- Player — Library использует Player для сценариев воспроизведения.
- Tracklists — Library использует Tracklists для сценариев, связанных с пользовательскими треклистами.
- Batch Operations — Library использует Batch Operations для сценариев над несколькими выбранными треками.
- Sheets — Library использует Sheets для временных пользовательских сценариев.
- Errors and Toasts — Library использует Errors and Toasts для пользовательской обратной связи.

# Контракты

- Library отвечает за пользовательский сценарий фонотеки: подключение музыкальных папок, просмотр структуры фонотеки, просмотр содержимого папок и запуск действий над треками.
- Library владеет только своим feature-flow и не становится владельцем архитектурных подсистем, которые использует.
- Library отображает готовое состояние и передаёт пользовательские намерения в соответствующие архитектурные контракты.
- Library использует Track Identity для действий, относящихся к физическому треку.
- Library получает актуальные данные трека через Runtime Metadata.
- Library инициирует сценарии воспроизведения, не становясь владельцем playback-состояния.
- Library инициирует действия с пользовательскими треклистами, не становясь владельцем их состава.
- Library инициирует массовые действия через Batch Operations.
- Library инициирует временные сценарии через Sheets и не делает их частью основного feature-flow.
- Library передаёт ошибки, предупреждения и успешные результаты в контракт пользовательской обратной связи.

# Ограничения

- Library не описывает и не определяет внутреннюю реализацию загрузки фонотеки.
- Library не становится владельцем runtime-данных трека.
- Library не становится владельцем устойчивой идентичности трека.
- Library не становится владельцем playback-состояния.
- Library не становится владельцем состава пользовательских треклистов.
- Library не создаёт собственную независимую систему навигации.
- Library не превращает временные сценарии в постоянную часть feature-flow.
- Library не определяет способ представления пользовательской обратной связи.
