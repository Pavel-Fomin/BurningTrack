# Назначение

Player — feature-подсистема пользовательских сценариев воспроизведения BurningTrack.

Она решает задачу управления текущим воспроизведением: пользователь видит текущий список воспроизведения, запускает или приостанавливает трек, меняет порядок треков, удаляет треки из списка и запускает связанные действия над выбранным треком или всем списком.

Player не описывает внутреннее устройство воспроизведения. Он использует существующие архитектурные контракты и остаётся feature-подсистемой пользовательского сценария playback.

# Пользовательский сценарий

Пользователь открывает Player, чтобы управлять текущим воспроизведением.

В Player пользователь выбирает трек для воспроизведения, ставит воспроизведение на паузу или продолжает его, меняет порядок треков, удаляет отдельные треки или очищает текущий список воспроизведения.

Для отдельного трека пользователь может открыть подробную информацию, перейти к треку в Library, запустить перемещение или переименование. Для всего текущего списка пользователь может инициировать сохранение как пользовательский Tracklist или экспорт.

Если действие требует временного пользовательского сценария, Player использует соответствующий presentation-контракт. Если действие завершается успешно или с ошибкой, Player использует контракт пользовательской обратной связи для результата.

# Основной поток

Типовой поток Player:

```text
Пользователь открывает Player
↓
Player показывает текущий список воспроизведения
↓
Пользователь выбирает действие над треком или списком
↓
Player инициирует соответствующий пользовательский сценарий
↓
Результат возвращается через соответствующий архитектурный контракт
↓
Player обновляет отображаемое состояние
```

Поток не фиксирует внутреннее устройство используемых архитектурных контрактов. Эти вопросы раскрываются в каноничных архитектурных документах.

# Участвующие подсистемы

- Root Screens — Player использует Root Screens для подключения как самостоятельной feature-подсистемы и применения общего screen-flow контракта.
- Navigation — Player использует Navigation для переходов из Player к связанным пользовательским сценариям.
- Runtime Metadata — Player использует Runtime Metadata для отображения актуальных данных трека.
- Track Identity — Player использует Track Identity для действий, относящихся к физическому треку.
- Sheets — Player использует Sheets для временных пользовательских сценариев.
- Errors and Toasts — Player использует Errors and Toasts для пользовательской обратной связи.
- Mini Player — Player использует Mini Player для общей playback presentation вне основного экрана Player.
- Library — Player использует Library для перехода к треку в фонотеке.
- Tracklists — Player использует Tracklists для сценариев, связанных с пользовательскими треклистами.
- Export — Player использует Export для экспорта текущего списка воспроизведения.
- Track Edit Session — Player использует Track Edit Session для временных сценариев изменения данных трека.

# Контракты

- Player отвечает за пользовательский сценарий воспроизведения и действия с текущим списком воспроизведения.
- Player отображает готовое состояние текущего списка воспроизведения и пользовательские действия над ним.
- Player инициирует запуск, паузу и выбор трека для воспроизведения.
- Player инициирует изменение порядка, удаление отдельных треков и очистку текущего списка воспроизведения.
- Player использует Track Identity для действий, относящихся к физическому треку.
- Player использует Runtime Metadata для актуальных данных трека.
- Player инициирует временные сценарии через Sheets.
- Player инициирует сценарии, связанные с пользовательскими треклистами, через Tracklists.
- Player инициирует экспорт текущего списка воспроизведения через Export.
- Player сообщает результаты пользовательских действий через контракт пользовательской обратной связи.

# Ограничения

- Player не описывает внутреннюю реализацию воспроизведения.
- Player не работает напрямую с источниками данных треков.
- Player не изменяет архитектурные контракты других подсистем.
- Player не превращает временные пользовательские сценарии в часть своего основного feature-flow.
- Player не определяет способ представления пользовательской обратной связи.
