# Назначение

Track Identity — это архитектурный контракт, который определяет, что считается одним и тем же треком в BurningTrack.

В рамках этого контракта трек — это физический музыкальный объект, который может участвовать в разных пользовательских и системных сценариях независимо от того, где и сколько раз он представлен в UI.

Track Identity нужен, чтобы все подсистемы ссылались на один и тот же трек одинаково, а конкретные UI-вхождения могли оставаться независимыми друг от друга.

# Проблема

Если физический файл, строка UI и пользовательский сценарий используют одну и ту же идентичность, приложение теряет возможность различать разные архитектурные вопросы.

Это создаёт несколько проблем:

- один и тот же трек нельзя безопасно добавить несколько раз в очередь или треклист;
- состояние конкретной строки может смешиваться с состоянием физического трека;
- удаление или выделение одной строки может ошибочно затронуть другие вхождения того же трека;
- операции над физическим треком начинают зависеть от UI-идентификаторов;
- переименование или перемещение файла может выглядеть как создание нового трека;
- feature-подсистемы начинают вводить собственные правила идентичности.

Единый контракт Track Identity отделяет устойчивую идентичность физического трека от идентичности конкретного отображаемого вхождения.

# Архитектурная схема

Место Track Identity в архитектуре:

```text
Physical File
↓
Track Identity
↓
┌────────────────────┬──────────────────┬───────────────┬──────────────┐
│                    │                  │               │              │
File Access          Runtime Metadata   Library         Player         Tracklists
│                    │                  │               │              │
└────────────────────┴──────────────────┴───────────────┴──────────────┘
↓
Feature State
↓
UI / Row Identity
```

Track Identity связывает физический трек с нижележащими операциями над файлом и с feature-подсистемами. Feature State может создавать отдельные row/display identity для конкретных вхождений, но эти идентификаторы не заменяют Track Identity.

# Основные компоненты

## Физический трек

Физический трек — это музыкальный файл как объект, с которым работают сценарии, относящиеся к самому файлу и его runtime-данным.

Физический трек не равен строке UI. Один физический трек может иметь несколько отображаемых вхождений в разных сценариях.

## Track Identity

Track Identity — устойчивая идентичность физического трека внутри приложения.

Она используется во всех сценариях, которым нужно обратиться к самому физическому треку, его файлу или runtime-данным.

## Row / Display Identity

Row / Display Identity — идентичность конкретного отображаемого вхождения трека.

Она используется для UI-сценариев, которым важно конкретное отображаемое вхождение трека, а не только физический трек.

## Слой разрешения Track Identity

Слой разрешения Track Identity отвечает за то, чтобы обнаруженный или импортированный физический трек получил устойчивую идентичность.

Этот слой не является UI-слоем и не зависит от metadata. Изменение тегов, обложки или длительности не должно создавать новую Track Identity.

## Runtime Metadata

Runtime Metadata связывает актуальные данные трека с Track Identity.

Runtime Metadata не создаёт собственную идентичность и не заменяет Track Identity. Он использует Track Identity как ключ связи с физическим треком.

## Feature State

Feature State связывает Track Identity с конкретным пользовательским сценарием.

Если сценарию нужно различать отдельные вхождения одного трека, Feature State использует row/display identity. Если сценарию нужно обратиться к физическому треку, он использует Track Identity.

## UI

UI отображает готовое состояние и работает с идентичностью на уровне пользовательского действия.

UI не создаёт Track Identity и не должен использовать собственную идентичность строки для файловых, metadata, playback или export-сценариев.

# Поток данных

Жизненный цикл использования Track Identity:

```text
Обнаружение или импорт физического трека
↓
Назначение или повторное использование Track Identity
↓
Связывание Track Identity с доступом к файлу
↓
Использование Track Identity в Runtime Metadata
↓
Использование Track Identity в feature-подсистемах
↓
Создание row/display identity для конкретных UI-вхождений
↓
Отображение в UI
```

Когда пользователь действует с конкретной строкой, сценарий сначала работает с row/display identity, если нужно выбрать или изменить конкретное вхождение. Если действие должно затронуть физический трек, сценарий переходит к Track Identity.

# Контракты

- Track Identity является каноничной устойчивой идентичностью физического трека внутри BurningTrack.
- Row / Display Identity не заменяет Track Identity и используется только для конкретных отображаемых вхождений.
- Один физический трек может иметь несколько row/display identity в разных feature-подсистемах.
- Все feature-подсистемы используют Track Identity, когда действие относится к физическому треку.
- Runtime Metadata связывается с треком через Track Identity.
- Изменение metadata, artwork или длительности не меняет Track Identity.
- Изменение имени файла или расположения файла не должно нарушать Track Identity, если трек остаётся тем же физическим объектом в пользовательском сценарии.
- Все сценарии, работающие с физическим треком, используют Track Identity.
- Все сценарии, работающие с конкретным отображаемым вхождением, используют row/display identity.
- Screen, ViewModel и UI не являются владельцами Track Identity.

# Инварианты

- Feature-подсистемы не вводят собственную устойчивую идентичность физического трека.
- Runtime Metadata не создаёт и не изменяет Track Identity.
- Row / Display Identity нельзя использовать для сценариев, работающих с физическим треком, его файлом или runtime-данными.
- Track Identity нельзя использовать как идентичность строки там, где один трек может иметь несколько вхождений.
- Пользовательские действия, начавшиеся со строки UI, должны переходить к Track Identity перед операцией над физическим треком.
- Внутреннее изменение способа хранения, поиска или восстановления файла не должно менять архитектурный контракт Track Identity.
- Screen и UI не становятся источником Track Identity.

# Зависимости

Track Identity используется документами и будущими каноничными описаниями следующих подсистем:

- Overview — место Track Identity в общей архитектуре BurningTrack.
- Root Screens — правила передачи пользовательских действий из UI в нижележащие подсистемы.
- Runtime Metadata — связь актуальных metadata с физическим треком.
- Library — идентичность треков фонотеки.
- Player — различение физического трека и вхождения в очереди.
- Tracklists — различение физического трека и строки треклиста.
- Batch Operations — применение массовых действий к выбранным физическим трекам.
- Export — доступ к физическим файлам через Track Identity.
