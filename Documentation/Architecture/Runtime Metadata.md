# Назначение

Runtime Metadata — это общий архитектурный контракт актуальных metadata трека во время работы приложения.

Он нужен, чтобы feature-подсистемы получали готовые данные трека через единый runtime-слой, а не читали файл, постоянное хранилище или внешние интеграции напрямую.

Runtime Metadata занимает место между нижележащими источниками данных и экранным State. Он не является UI-состоянием, постоянным хранилищем или интеграцией чтения тегов.

# Проблема

Без отдельного runtime-контракта разные экраны начинают получать metadata разными способами: одни читают файл, другие используют сохранённые данные, третьи обращаются к техническому кэшу или внешней интеграции.

Это создаёт несколько проблем:

- UI начинает владеть файловым доступом и деталями чтения metadata;
- после изменения файла разные подсистемы могут показывать разные значения;
- feature-подсистемы становятся зависимыми от внутреннего устройства storage или внешних readers;
- технические кэши начинают восприниматься как публичный контракт;
- изменение способа чтения metadata требует изменений в экранных сценариях.

Runtime Metadata отделяет отображаемые runtime-данные от постоянного хранения и низкоуровневого чтения файла.

# Архитектурная схема

Место Runtime Metadata в общей архитектуре:

```text
Storage / Track Identity
↓
File Access / Metadata Reading
↓
Runtime Metadata
↓
Feature State
↓
UI
```

Storage и Track Identity помогают найти трек и его файл. Слой чтения metadata получает данные из файла или внешней интеграции. Runtime Metadata превращает эти данные в единый runtime-контракт, который feature-подсистемы используют для сборки State и обновления UI.

При изменении файла, тегов, обложки или доступности данные должны возвращаться наверх через тот же runtime-контракт:

```text
Изменение данных трека
↓
Обновление Runtime Metadata
↓
Распространение Runtime Metadata
↓
Feature State
↓
UI
```

# Основные компоненты

## Источник данных

Источник данных отвечает за устойчивую идентичность трека, путь к файлу и сохранённое состояние библиотеки.

Он не является экранным источником metadata и не должен подменять Runtime Metadata для данных, прочитанных из файла во время работы приложения.

## Слой чтения metadata

Слой чтения metadata получает сырые данные из файла или внешней интеграции.

Он находится ниже runtime-контракта и не является публичным API для UI или экранных сценариев.

## Слой формирования Runtime Metadata

Слой формирования Runtime Metadata собирает готовое runtime-представление трека из устойчивой идентичности, файлового доступа и прочитанных metadata.

Он не обновляет UI, не принимает пользовательские решения и не является постоянным хранилищем.

## Runtime Snapshot

Runtime Snapshot — готовое runtime-представление актуальных metadata одного трека.

Он является UI-facing контрактом для metadata, доступности, имени файла, длительности и обложки в том виде, в котором они нужны feature-сценариям.

## Runtime-хранилище

Runtime-хранилище удерживает последние собранные runtime-представления для повторного использования внутри текущей работы приложения.

Оно не читает файлы само и не является persistence-слоем. Его содержимое можно пересобрать без изменения архитектурного контракта.

## Слой распространения Runtime Metadata

Слой распространения Runtime Metadata доставляет feature-подсистемам готовое runtime-представление после изменения данных трека.

Подписчики получают уже сформированную runtime-модель и не должны самостоятельно перечитывать файл, чтобы подтвердить изменение.

## Screen / ViewModel

Screen и ViewModel используют Runtime Metadata как вход для сборки State.

Они могут решать, как представить данные пользователю, но не должны становиться владельцами чтения metadata, файлового доступа или внешней интеграции.

# Поток данных

Базовый поток получения metadata:

```text
Track Identity
↓
File Access / Metadata Reading
↓
Runtime Metadata
↓
State
↓
UI
```

Поток после изменения данных:

```text
Изменение файла, тегов, обложки или доступности
↓
Обновление Runtime Metadata
↓
Готовое runtime-представление
↓
Обновление feature State
↓
Обновление UI
```

Feature-подсистема может использовать базовые данные из storage до появления готового runtime-представления. Но metadata, прочитанная из файла или обновлённая после изменения трека, должна попадать в экранный State через Runtime Metadata.

# Контракты

- Runtime Metadata является каноничным UI-facing контрактом для актуальных metadata трека во время работы приложения.
- UI не читает metadata напрямую из файла.
- UI не обращается напрямую к низкоуровневым readers, внешним интеграциям или техническим metadata-кэшам.
- Runtime Metadata отделён от постоянного storage и не является persistence-слоем.
- Изменение файла, тегов, обложки, имени файла или доступности должно приводить к обновлению Runtime Metadata.
- После изменения данных feature-подсистемы получают готовое runtime-представление, а не только сигнал о том, что трек изменился.
- Runtime Metadata связывается с треком через устойчивую Track Identity.
- Feature State строится поверх Runtime Metadata и не должен подменять его собственным чтением metadata.
- Runtime Metadata не определяет playback-flow, navigation, membership треклистов или batch-сценарий; он предоставляет актуальные данные трека для этих подсистем.

# Инварианты

- Runtime-контракт metadata одинаков для Library, Player, Tracklists, Sheets и Batch Operations.
- Низкоуровневое чтение metadata остаётся ниже Runtime Metadata.
- Технические кэши можно очистить, заменить или пересобрать без изменения UI-контракта.
- Изменение реализации чтения metadata не должно менять контракт Runtime Metadata.
- Изменения данных трека не должны обходить слой распространения Runtime Metadata.
- Runtime Metadata не хранится как постоянный источник правды и может быть пересобран из нижележащих источников.
- UI не становится владельцем файлового доступа или metadata-reading flow.

# Зависимости

Runtime Metadata используется документами и будущими каноничными описаниями следующих подсистем:

- Overview — место Runtime Metadata в общей архитектуре BurningTrack.
- Root Screens — правила подключения runtime-данных к экранному State.
- Track Identity — идентификатор, через который Runtime Metadata связывается с физическим треком.
- Library — использование актуальных metadata в фонотеке.
- Player — использование актуальных metadata в playback-сценариях.
- Tracklists — использование актуальных metadata в строках треклистов.
- Batch Operations — использование готового runtime-представления для массовых операций.
- Track Edit Session — обновление Runtime Metadata после изменения tags, artwork или имени файла.
- TagLib — внешняя интеграция, находящаяся ниже runtime-контракта.
