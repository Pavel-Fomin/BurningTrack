# Назначение

SQLite — единственное постоянное хранилище внутреннего рабочего состояния BurningTrack.

Слой SQLite нужен, чтобы приложение могло восстанавливать своё состояние после перезапуска, выполнять изменения атомарно и держать внутренние данные в одном устойчивом месте. Верхние слои приложения не должны знать, как эти данные физически сохранены.

JSON не используется как рабочее постоянное хранилище приложения. JSON-файлы не должны становиться параллельным источником состояния, резервной рабочей копией или способом восстановления данных приложения.

Сами музыкальные файлы остаются внешними объектами файловой системы. SQLite хранит рабочее состояние приложения, связанное с этими объектами: устойчивые идентификаторы, доступ, пользовательские списки, настройки и другие данные, которые приложение должно восстановить без обращения к другому постоянному хранилищу.

# Источник истины

Для рабочих данных приложения действует принцип единственного источника истины: каждое постоянное состояние хранится в одном постоянном месте.

Каждая сущность имеет только одного владельца постоянного состояния. SQLite хранит состояние сущности, а не несколько его постоянных копий в разных местах.

Если данные принадлежат SQLite-слою, то именно SQLite является источником истины для их восстановления. Runtime-состояние, кэши, опубликованный экранный State и временные модели могут существовать только как производные представления этих данных или как данные текущей сессии.

Недопустимы:

- несколько рабочих постоянных копий одних и тех же данных;
- дублирование состояния между независимыми persistent-хранилищами;
- параллельное использование JSON, файловых снимков или других рабочих хранилищ для тех же данных;
- восстановление рабочего состояния из runtime-кэша вместо SQLite.

Если пользовательская операция завершилась успешно, состояние приложения после этой операции должно быть восстановимо из SQLite. Для операций, которые меняют внешний файл, SQLite должен хранить обновлённое состояние приложения, связанное с этим файлом.

# Архитектурные слои

С точки зрения SQLite поток зависимости направлен сверху вниз:

```text
View
↓
ViewModel
↓
Manager / Domain
↓
Database Store
↓
SQLite
```

Если feature использует `Action` и `ActionHandler`, они находятся между `ViewModel` и `Manager / Domain`: принимают пользовательское намерение и вызывают нужный сценарий, но не становятся persistence-слоем.

Ответственность слоёв:

- `View` отображает готовое состояние и передаёт пользовательские действия выше по feature-flow.
- `ViewModel` собирает экранное состояние и преобразует действия пользователя в команды текущего сценария.
- `Manager / Domain` выполняет бизнес-сценарий, координирует подсистемы и обращается к постоянному хранилищу только через Store.
- `Database Store` является границей SQLite-слоя для прикладного кода.
- `SQLite` хранит постоянные данные приложения и выполняет операции, которые Store формирует для него.

Нижележащая инфраструктура SQLite состоит из владельца базы, исполнителя операций, соединения, мигратора и обёрток над низкоуровневым API. Эти компоненты относятся к persistence-слою и не должны становиться зависимостью UI или feature-flow.

# Database Store

Database Store скрывает физическое хранение от остального приложения.

Любой сценарий изменения постоянного состояния проходит через Store. Store является единственной точкой создания, изменения и удаления постоянных данных приложения.

Store отвечает за:

- преобразование между моделями приложения и моделями хранения;
- выбор минимального набора данных, нужного сценарию;
- выполнение чтения и записи через общий исполнитель базы;
- объединение связанных изменений в транзакции;
- сохранение инвариантов, которые относятся к постоянному хранению;
- преобразование технических ошибок хранения в контракт, понятный верхнему слою.

Для верхних слоёв только Store знает структуру хранения, SQL, транзакции и особенности SQLite. Физически SQL может быть вынесен в общие query-определения persistence-слоя, но использовать его должны Store и инфраструктура SQLite, а не UI, ViewModel, ActionHandler, Manager или произвольный Domain-код.

В текущей реализации есть два уровня Store:

- фасады Store, которые отдают наружу модели приложения и закрывают детали хранения от Manager / Domain;
- низкоуровневые SQLite Store, которые работают с row-моделями и выполняют конкретные операции хранения.

Это внутреннее разделение не меняет внешний контракт: прикладной код работает через публичный Store API и не обращается к SQLite напрямую.

# Масштабирование

Новая функциональность должна встраиваться в существующую цепочку:

```text
View
↓
ViewModel
↓
Manager / Domain
↓
Database Store
↓
SQLite
```

При добавлении новой сущности или сценария сначала определяется владелец ответственности: feature, manager/domain-сервис и Store. Затем постоянное состояние добавляется в SQLite через Store и миграции схемы.

Новая функциональность должна расширять существующую архитектуру, а не создавать параллельный способ хранения или альтернативную цепочку доступа к данным. Направление зависимостей сохраняется: верхние слои формируют намерение и бизнес-сценарий, а постоянное состояние меняется только через Store.

Новая функциональность не должна требовать обращения к SQLite из View, ViewModel или ActionHandler. Если сценарию нужны новые данные, для него расширяется существующий Store или добавляется новый Store с понятной зоной ответственности.

Расширение SQLite-слоя не должно менять направление зависимостей. Верхние слои продолжают работать с бизнес-моделями и командами, а не с row-моделями, SQL или деталями соединения.

# Независимость модулей

Модули должны быть изолированы друг от друга на уровне хранения.

Каждый модуль работает со своими постоянными данными через Store, который является его persistence-контрактом. Модуль не должен читать или изменять внутреннюю структуру хранения другого модуля.

Если сценарий требует данных другой подсистемы, он должен использовать публичный Manager / Domain-контракт или публичный Store-фасад этой подсистемы. Прямое обращение к низкоуровневым Store другой области, к query-определениям или к row-моделям нарушает изоляцию.

Общие данные допускаются только через явно выделенный общий Store или общий доменный контракт. Такое расширение должно фиксировать владельца данных и не создавать вторую постоянную копию того же состояния.

# Расширяемость

SQLite-слой можно развивать без изменения архитектурных принципов.

Развитие модели хранения должно идти через последовательную эволюцию существующей модели. Новые таблицы появляются тогда, когда новая сущность действительно требует собственной области ответственности, а не как способ обойти существующие Store или разложить одно состояние на независимые постоянные копии.

Допускается:

- расширять схему SQLite;
- добавлять новые Store;
- добавлять новые миграции схемы SQLite;
- менять внутреннее устройство Store;
- оптимизировать запросы и транзакции внутри persistence-слоя.

При этом не меняются базовые правила:

- SQLite остаётся единственным рабочим постоянным хранилищем приложения;
- верхние слои не получают прямой доступ к SQLite;
- Store остаётся границей между приложением и физическим хранением;
- данные не дублируются в параллельных persistent-хранилищах;
- миграции изменяют схему, но не становятся местом бизнес-сценариев.

# Производительность

Производительность должна достигаться внутри persistence-слоя. Оптимизация не должна нарушать архитектурные границы или переносить задачи хранения в UI, runtime-кэш, ViewModel или другие верхние слои приложения ради скорости.

Основные правила:

- загружать только данные, необходимые текущему сценарию;
- выполнять массовые изменения транзакциями;
- избегать повторных обращений к базе, если данные уже получены в рамках того же сценария;
- не поднимать большие наборы данных в память без необходимости;
- держать сортировку, фильтрацию и атомарность в persistence-слое, если это естественно для сценария;
- не использовать runtime-кэш как замену корректной модели хранения.

Производительная реализация не должна нарушать архитектурные границы. Оптимизация SQL, транзакций и структуры хранения остаётся внутри SQLite-слоя.

# Надёжность

После успешного завершения операции рабочее состояние приложения должно быть восстановимо из SQLite.

Пользовательская операция считается успешно завершённой только после успешного сохранения её постоянного состояния.

Если операция меняет несколько связанных частей постоянного состояния, Store должен сохранять их атомарно. Частично применённое состояние не должно считаться успешным результатом пользовательского сценария.

Runtime-кэш, экранный State и временные модели могут ускорять работу текущей сессии, но они не являются источником восстановления. Их можно очистить и пересобрать без потери успешных пользовательских изменений.

Ошибки SQLite должны доходить до уровня, который может корректно завершить сценарий: показать ошибку, откатить runtime-состояние или не подтверждать успешное действие пользователю.

# Ограничения

Основное ограничение: SQL не должен выполняться вне persistence-слоя. Остальные правила являются следствиями этой границы.

Запрещено:

- использовать JSON как рабочее постоянное хранилище;
- обращаться к SQLite из `View`;
- обращаться к SQLite из `ViewModel`;
- обращаться к SQLite из `ActionHandler`;
- обращаться к SQLite напрямую из `Manager` или произвольного `Domain`-кода;
- импортировать `SQLite3` вне SQLite-инфраструктуры;
- использовать `DatabaseConnection`, `DatabaseExecutor`, statement-обёртки или row-reader вне persistence-слоя;
- выполнять SQL из UI, ViewModel, ActionHandler, Manager или feature-кода;
- передавать row-модели SQLite в экранный State как публичный контракт;
- создавать второе рабочее persistent-хранилище для данных, которые уже принадлежат SQLite.

Текущие допустимые исключения:

- миграции схемы выполняют SQL внутри SQLite-инфраструктуры;
- владелец базы настраивает соединение SQLite при старте приложения;
- исполнитель базы управляет командами транзакций;
- DEBUG-диагностика может выполнять read-only агрегаты внутри persistence-слоя и не участвует в production-поведении;
- некоторые Manager / Domain-компоненты могут получать или хранить `AppDatabase` только как зависимость для создания Store, но не должны извлекать из него executor, соединение или выполнять SQL напрямую;
- доменный код может сохранять данные через публичный Store API, если это часть текущего архитектурного сценария и не раскрывает SQL или row-модели выше Store.

Эти исключения не разрешают обходить Store из feature-кода. Они относятся к инфраструктуре SQLite или к существующим сценариям, которые всё равно проходят через публичный Store API.
