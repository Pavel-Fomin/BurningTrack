# Назначение

Errors and Toasts — это архитектурный контракт пользовательской обратной связи в BurningTrack.

Он определяет, как результат пользовательского действия, ошибка или предупреждение превращаются в понятное пользователю feedback-событие и как это событие попадает в UI без смешивания с бизнес-логикой.

Документ описывает не конкретный механизм Toast, а устойчивую архитектурную роль обратной связи: отделить причину результата от способа его показа.

# Проблема

Если UI самостоятельно показывает ошибки, feature-подсистемы начинают зависеть от конкретного способа отображения.

Это создаёт несколько проблем:

- разные сценарии показывают ошибки и успешные действия несогласованно;
- бизнес-логика начинает выбирать тексты, визуальный стиль или время показа;
- одна и та же ошибка может отображаться по-разному в разных feature-flow;
- успешный результат и ошибка проходят разными путями и теряют единый lifecycle;
- UI становится владельцем ошибок вместо отображения готового состояния;
- замена способа показа уведомлений требует изменений в feature-подсистемах.

Единый feedback-контракт нужен, чтобы feature-flow сообщал о результате действия, а presentation-слой отдельно решал, как показать этот результат пользователю.

# Архитектурная схема

Место пользовательской обратной связи в архитектуре BurningTrack:

```text
Пользовательское действие
↓
Feature Flow / Command Flow
↓
Результат выполнения
↓
Error Contract / Feedback Contract
↓
Presentation State
↓
Presentation
↓
UI
```

Для ошибок и успешных действий используется один общий контур:

```text
Ошибка или успешный результат
↓
Feedback Event
↓
Presentation State
↓
UI feedback
```

Схема фиксирует архитектурные роли. Она не описывает визуальную реализацию уведомления или конкретный UI-механизм.

# Основные компоненты

## Error Contract

Error Contract описывает пользовательски значимую причину сбоя.

Он отделяет техническую причину ошибки от того, что должно быть показано пользователю. Error Contract не показывает уведомление, не меняет navigation и не выполняет fallback-логику.

## Feedback Contract

Feedback Contract описывает пользовательское событие обратной связи.

Это может быть ошибка, предупреждение или успешное завершение действия. Feedback Contract хранит смысл события, но не является визуальным представлением и не выполняет бизнес-операции.

## Формирование feedback

Формирование feedback переводит ошибку или результат выполнения в пользовательское feedback-событие.

Эта роль является границей между причиной результата и presentation-состоянием. Она не читает feature-состояние сверх необходимого контекста, не выполняет системные операции и не показывает UI самостоятельно.

## Presentation State

Presentation State — готовая модель пользовательской обратной связи для отображения.

Она содержит только данные, необходимые для показа пользователю. Presentation State не знает источник события, не обращается к managers, storage, Runtime Metadata или внешним интеграциям.

# Поток данных

Типовой поток ошибки:

```text
Пользовательское действие
↓
Feature Flow / Command Flow
↓
Ошибка пользовательского уровня
↓
Error Contract
↓
Feedback Contract
↓
Presentation State
↓
Presentation
↓
UI
```

Типовой поток успешного действия:

```text
Пользовательское действие
↓
Feature Flow / Command Flow
↓
Успешный результат
↓
Feedback Contract
↓
Presentation State
↓
Presentation
↓
UI
```

Если feedback относится к данным трека, необходимые данные должны приходить через существующие runtime и identity-контракты. Feedback не должен сам читать файлы, пересобирать feature-состояние или становиться источником данных для других подсистем.

# Контракты

- Пользовательская обратная связь формируется отдельно от UI.
- UI отображает готовое feedback-состояние и не становится владельцем ошибок.
- Feature Flow сообщает о результате действия через feedback-контракт, а не через прямое управление визуальным представлением.
- Command Flow может формировать feedback после успешного завершения действия, если он владеет итогом пользовательской команды.
- Ошибка пользовательского уровня проходит через Error Contract перед показом пользователю.
- Представление ошибки отделено от её причины.
- Ошибка, предупреждение и успешное завершение проходят через единый feedback-контракт.
- Presentation не принимает бизнес-решения и не выполняет системные операции.
- Feedback не подменяет Runtime Metadata, Track Identity, Navigation или Sheet Flow.
- Feedback-событие не используется как способ передачи данных между подсистемами.

# Инварианты

- UI не становится владельцем ошибок или успешных результатов.
- Feature-подсистема не зависит от конкретного способа отображения feedback.
- Замена визуального способа показа уведомлений не должна менять архитектурный feedback-контракт.
- Ошибка не меняет Navigation сама по себе.
- Toast или другое уведомление не становится способом передачи данных между подсистемами.
- Presentation State можно заменить или пересобрать без изменения причины ошибки или результата действия.
- Бизнес-логика не выбирает способ показа feedback.
- Ошибка пользовательского уровня не должна завершаться молча, если пользователь ожидает результат действия.

# Зависимости

Errors and Toasts используются документами и будущими каноничными описаниями следующих подсистем:

- Overview — место пользовательской обратной связи в общей архитектуре BurningTrack.
- Root Screens — передача пользовательских действий из UI в feature-flow и возврат feedback.
- Navigation — отделение feedback от переходов между сценариями.
- Sheets — feedback временных presentation-flow.
- Library — ошибки и успешные результаты фонотеки.
- Player — ошибки и успешные результаты playback-сценариев.
- Tracklists — ошибки и успешные результаты сценариев треклистов.
- Batch Operations — итоговая обратная связь массовых действий.
- Export — обратная связь сценариев экспорта.
