# Назначение

Этот документ фиксирует каноничный архитектурный паттерн корневых экранов BurningTrack.

Единый паттерн нужен, чтобы каждый основной раздел приложения строился одинаково: экран задаёт границу раздела, отображение получает готовое состояние, пользовательские намерения передаются как действия, а системные изменения выполняются вне UI.

Документ описывает общий подход к построению корневого экрана. Он не описывает отдельные возможности Library, Player, Tracklists или Settings.

# Что считается корневым экраном

Корневой экран — это самостоятельная точка входа пользовательского сценария, которая владеет собственным screen-flow.

Архитектурно это означает, что экран определяет границу сценария, состояние для отображения, пользовательские действия, зависимости flow и правила взаимодействия с нижележащими подсистемами.

# Проблема

Если каждый корневой экран строится по собственным правилам, проект быстро теряет единый архитектурный контракт:

- бизнес-логика начинает попадать в UI;
- пользовательские действия обрабатываются в разных местах;
- состояние экрана собирается рядом с отображением;
- зависимости становятся неявными;
- сложнее понять, где добавлять новый сценарий;
- сложнее безопасно менять внутреннюю реализацию без регрессий.

Единый контракт нужен, чтобы новый или изменяемый корневой экран было можно проверить по одной модели ответственности, а не по локальным привычкам конкретного раздела.

# Архитектурная схема

Типовая схема взаимодействия компонентов корневого экрана:

```text
Screen
↓
ViewModel
↓
Слой формирования состояния
↓
State
↓
UI

UI
↓
Action
↓
Слой обработки пользовательских действий
↓
Managers / Coordinators / Presenters
↓
ViewModel
```

Конкретная реализация может отличаться, но архитектурные контракты остаются неизменными: Screen является границей раздела и собирает flow экрана, UI отображает готовый State и отправляет Action, ViewModel синхронизирует экранное состояние с системными источниками, слой обработки пользовательских действий выполняет действия через нижележащие подсистемы, а слой сборки зависимостей собирает production-зависимости.

В текущей реализации эти роли могут быть представлены отдельными типами (например StateBuilder, ActionHandler, Factory) либо объединены внутри других компонентов при сохранении архитектурных контрактов.

# Основные компоненты

## Screen

Screen является входной точкой корневого раздела приложения.

Он отвечает за сборку flow экрана, подключение экранной ViewModel, передачу State в UI и маршрутизацию пользовательских действий в обработчик.

Screen не является местом для бизнес-логики, файловых операций, playback-логики или принятия системных решений.

## UI

UI отображает готовое состояние и сообщает о пользовательских намерениях наружу.

UI не должен читать managers, registries, coordinators или другие системные источники напрямую. Если данным нужно попасть на экран, они должны прийти через State.

## State

State описывает готовую модель экрана для отображения.

Он содержит данные, флаги и представления строк, которые уже подготовлены для UI. State не выполняет действия и не знает, из каких системных источников он был собран.

## Action

Action описывает намерение пользователя на языке экрана.

Action не выполняет работу самостоятельно и не содержит ссылок на системные зависимости. Его задача — передать смысл пользовательского действия из UI в слой обработки.

## ViewModel

ViewModel является экранным orchestration-слоем.

Она хранит или публикует готовый State, синхронизирует его с релевантными источниками приложения и передаёт пользовательские действия дальше. ViewModel не должна заменять менеджеры, сервисы или координаторы нижнего уровня.

## Слой формирования состояния

Слой формирования состояния преобразует данные подсистем в State.

Он отвечает только за подготовку состояния для экрана. Он не выполняет пользовательские действия, не управляет навигацией и не меняет системное состояние.

## Слой обработки пользовательских действий

Слой обработки пользовательских действий выполняет пользовательские действия, пришедшие из UI.

Он делегирует работу managers, coordinators, presenters или специализированным обработчикам. Этот слой не отображает UI и не является источником готового State.

## Слой сборки зависимостей

Слой сборки зависимостей собирает production-зависимости экрана.

Этот слой отделяет создание зависимостей от UI и делает границы flow явными. Дочерние UI-компоненты не должны знать, как собирается production-граф экрана.

# Поток данных

Типовой жизненный цикл пользовательского действия:

```text
Пользовательское действие
↓
UI отправляет Action
↓
Слой обработки пользовательских действий обрабатывает действие
↓
Нижележащая подсистема меняет системное состояние
↓
ViewModel получает актуальные данные
↓
Слой формирования состояния собирает новый State
↓
UI отображает новый State
```

Для действий без изменения системного состояния поток может завершиться на уровне слоя обработки пользовательских действий или презентационного слоя. Для действий, влияющих на экран, UI должен обновляться через новый State, а не через прямое изменение отображения внутри View.

# Контракты

- Screen задаёт границу корневого раздела и собирает flow экрана.
- Корневой экран является владельцем собственного screen-flow: определяет границу сценария, экранное состояние, пользовательские действия, зависимости flow и взаимодействие с нижележащими подсистемами.
- Screen не содержит бизнес-логику и не выполняет системные операции напрямую.
- UI отображает State и отправляет Action.
- UI не читает системные managers, registries и coordinators напрямую.
- Action описывает пользовательское намерение, но не выполняет действие.
- ViewModel подготавливает готовое состояние экрана и синхронизирует его с источниками приложения.
- Слой формирования состояния только строит State.
- Слой обработки пользовательских действий обрабатывает Action и делегирует работу нижележащим подсистемам.
- Слой сборки зависимостей собирает зависимости экрана вне дочернего UI.
- Компоненты screen-flow должны иметь одну основную ответственность.

# Инварианты

- Новый корневой экран должен следовать паттерну State → UI и UI → Action.
- Бизнес-логика не переносится в Screen или UI.
- Состояние для отображения не собирается внутри дочерних UI-компонентов из системных managers.
- Пользовательские намерения выходят из UI как Action или эквивалентный callback flow.
- Сборка State отделена от выполнения Action.
- Production-зависимости собираются на границе flow, а не внутри отображающих компонентов.
- Если корневой экран содержит несколько вложенных flow, каждый flow сохраняет собственные границы ответственности.
