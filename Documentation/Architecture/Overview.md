# Назначение

Overview — главный вход в архитектурную документацию BurningTrack.

Документ показывает, из каких архитектурных подсистем состоит приложение, как они связаны между собой и куда переходить за подробным описанием отдельного архитектурного вопроса.

С него начинается знакомство с архитектурой проекта, потому что он даёт общую карту без погружения в реализацию конкретных экранов, подсистем хранения или внешних интеграций.

# Проблема

Без единого архитектурного обзора крупный проект начинает восприниматься как набор отдельных экранов, компонентов и файлов.

Это создаёт несколько проблем:

- сложно понять, к какой подсистеме относится новая функциональность;
- разные части приложения начинают решать похожие задачи разными способами;
- архитектурные контракты теряются среди деталей реализации;
- документация начинает дублировать сама себя;
- разработчику приходится читать код нескольких подсистем, чтобы понять общую картину.

Overview фиксирует карту верхнего уровня и направляет читателя к каноничным документам по конкретным вопросам.

# Архитектурная схема

Высокоуровневая схема BurningTrack:

```text
Application Shell
↓
Root Screens and Navigation
↓
Feature Subsystems
↓
Shared Architectural Contracts
↓
Domain and Infrastructure
↓
Storage and Registries
↓
External Integrations
```

Верхние уровни отвечают за вход пользователя в приложение: application shell подключает глобальные presentation-контуры, а Root Screens and Navigation задают основные пользовательские маршруты.

Средний уровень состоит из feature-подсистем и общих архитектурных контрактов, которые связывают экраны, сценарии и runtime-состояние. Нижние уровни выполняют системные операции, хранят устойчивые данные и изолируют внешние технологии от остальной архитектуры.

# Основные подсистемы

## Root Screens

Root Screens задают границы основных пользовательских сценариев приложения. Их ключевая ответственность — владеть собственным screen-flow и связывать UI с нижележащими подсистемами.

Каноничный документ: [Root Screens](Root%20Screens.md).

## Navigation

Navigation отвечает за переходы между крупными разделами и маршруты внутри разделов. Его ключевая ответственность — хранить навигационное намерение отдельно от feature-логики.

Каноничный документ по Navigation будет создан отдельно.

## Library

Library отвечает за фонотеку и доступ к музыкальным файлам. Её ключевая ответственность — связать пользовательскую библиотеку с устойчивыми registry/storage-контрактами.

Каноничный документ по Library будет создан отдельно; отдельный аспект раскрыт в [Library loading architecture](../Legacy/Library%20loading%20architecture.md).

## Player

Player отвечает за пользовательские сценарии воспроизведения. Его ключевая ответственность — отделять playback-flow от остальных feature-подсистем.

Каноничный документ по Player будет создан отдельно.

## Tracklists

Tracklists отвечают за пользовательские списки треков. Их ключевая ответственность — хранить и изменять состав треклистов как отдельную feature-подсистему.

Каноничный документ по Tracklists будет создан отдельно.

## Runtime Metadata

Runtime Metadata отвечает за актуальные данные трека, которые используются экранами во время работы приложения. Ключевая ответственность — дать UI готовый runtime-контракт вместо прямого чтения файлов.

Каноничный документ: [Runtime Metadata](Runtime%20Metadata.md).

## Track Identity

Track Identity отвечает за устойчивую идентичность треков. Ключевая ответственность — отделить file identity от row/display identity.

Каноничный документ: [Track Identity](Track%20Identity.md).

## Batch Operations

Batch Operations отвечают за действия над несколькими выбранными треками. Их ключевая ответственность — отделить массовый пользовательский сценарий от одиночных операций.

Каноничный документ по Batch Operations будет создан отдельно; существующий контракт описан в [Multiselect actions](../Legacy/Multiselect%20actions.md).

## Export

Export отвечает за передачу выбранных треков за пределы приложения. Его ключевая ответственность — быть отдельным feature-flow, не смешанным с playback или tracklist-хранилищем.

Каноничный документ по Export будет создан отдельно.

## Sheets

Sheets отвечают за временные presentation-flow поверх основных экранов. Их ключевая ответственность — отделять модальные сценарии от root screen-flow.

Каноничный документ по Sheets будет создан отдельно.

## Errors and Toasts

Errors and Toasts отвечают за пользовательскую обратную связь. Их ключевая ответственность — централизовать представление ошибок и коротких status-сообщений.

Каноничный документ Errors and Toasts будет создан отдельно; существующий контракт описан в [Toast Architecture](../../TOAST_ARCHITECTURE.md).

## External Integrations

External Integrations отвечают за подключение внешних технологий. Их ключевая ответственность — изолировать внешние API от feature-flow и UI-слоя.

Каноничный документ текущей интеграции: [TagLib](../Integrations/TagLib.md).

# Поток данных

Общий поток данных в BurningTrack строится вокруг пользовательского намерения и обновления готового состояния для UI:

```text
Пользовательское действие
↓
Screen / feature-flow
↓
Domain / infrastructure flow
↓
Storage / registries / external integrations
↓
Runtime contract или событие изменения
↓
ViewModel / state composition
↓
State
↓
UI
```

Screen-flow принимает пользовательское намерение и передаёт его в соответствующий сценарий. После изменения системного состояния данные возвращаются наверх через общий runtime-контракт, событие изменения, обновлённый State или пользовательское feedback-событие.

UI не является источником системных решений. Он отображает готовое состояние и передаёт намерения пользователя в соответствующий flow.

# Контракты

- Каждая подсистема имеет собственную область ответственности.
- Подсистемы взаимодействуют через явные архитектурные контракты: State, Action, command-flow, runtime contracts, events, registries, adapters или protocols.
- Root Screens владеют своими screen-flow и не подменяют domain, storage или integration-слои.
- Feature flows не должны напрямую обращаться к низкоуровневым внешним интеграциям, если для этого существует проектный контракт.
- Runtime metadata является отдельным контуром и не смешивается с UI-состоянием или storage metadata.
- Track identity отделена от row identity и используется в file-facing сценариях.
- Storage, доступ к файлам и identity являются разными архитектурными ответственностями.
- Ошибки пользовательского уровня проходят через единый errors/toast-контракт.
- Каждый архитектурный вопрос должен иметь один каноничный документ; остальные документы ссылаются на него вместо дублирования.

# Инварианты

- Новая функциональность должна встраиваться в существующую архитектурную подсистему или явно вводить новую подсистему с собственной ответственностью.
- Новая подсистема не должна смешивать screen-flow, domain-сценарии, storage, runtime metadata и external integration в одном уровне ответственности.
- UI не должен становиться владельцем бизнес-логики, файлового доступа, persistence или низкоуровневых интеграций.
- Изменения, влияющие на отображаемые данные, должны возвращаться в UI через общий runtime/update контракт.
- Новые внешние интеграции должны быть изолированы за проектным контрактом.
- Архитектурная документация должна обновляться при изменении архитектурного контракта, а не при внутреннем рефакторинге реализации.

# Зависимости

Overview является верхним уровнем архитектурной документации и направляет к специализированным документам:

- Root Screens — правила построения корневых экранов и screen-flow.
- Navigation — правила маршрутизации и переходов.
- Runtime Metadata — контракт runtime snapshot и обновления данных трека.
- Track Identity — контракт file identity и row identity.
- Library — архитектура фонотеки и загрузки треков.
- Player — архитектура playback-сценариев.
- Tracklists — архитектура пользовательских треклистов.
- Batch Operations — архитектура массовых действий.
- Export — архитектура экспорта.
- Sheets — архитектура presentation-flow.
- Errors and Toasts — архитектура ошибок и пользовательской обратной связи.
- TagLib — архитектура внешней интеграции чтения и записи тегов.
