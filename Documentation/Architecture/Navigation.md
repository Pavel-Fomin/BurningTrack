# Назначение

Navigation — это архитектурный контракт переходов между пользовательскими сценариями BurningTrack.

Он определяет, какой слой владеет навигационным состоянием, где проходит граница root screen-flow, как feature-подсистемы открывают собственные сценарии и как временные presentation-flow подключаются поверх основных экранов.

Navigation определяет, кто владеет переходом и куда ведёт пользовательское намерение. Он не описывает визуальный способ показа перехода. Документ фиксирует только архитектурные роли и правила ответственности.

# Проблема

Если каждый экран самостоятельно управляет навигацией, приложение быстро теряет единый flow:

- дочерние UI-компоненты начинают открывать чужие сценарии напрямую;
- бизнес-логика смешивается с переходами между экранами;
- разные feature-подсистемы начинают создавать собственные правила маршрутизации;
- presentation-flow может начать менять структуру основной навигации;
- становится сложно понять, кто владеет текущим сценарием и куда добавлять новый переход;
- одноразовые навигационные намерения превращаются в постоянное состояние.

Единый навигационный контракт нужен, чтобы новый сценарий можно было отнести к правильному контексту: выбор root-раздела, root screen, feature или presentation.

# Архитектурная схема

Место Navigation в общей архитектуре BurningTrack:

```text
Application
↓
Navigation
↓
Root Screens
↓
Feature Navigation
↓
Feature State
↓
UI

Root Screens
↓
Presentation Navigation
↓
Presentation Flows
```

В текущей архитектуре Navigation состоит из нескольких ролей, но не каждый root-раздел обязан иметь собственную сложную навигацию. Если feature-подсистема не содержит вложенных экранов или маршрутов, её navigation-flow остаётся минимальным и ограничивается обработкой пользовательских действий или presentation-flow.

# Основные компоненты

## Root Navigation

Root Navigation принадлежит корневому экрану.

Root Screen определяет границу своего navigation-flow, подключает нужное состояние и передаёт пользовательские действия в нижележащие обработчики. Root Screen не управляет navigation-flow других корневых разделов.

## Feature Navigation

Feature Navigation отвечает за переходы внутри одной feature-подсистемы.

Если feature-подсистема содержит собственные сценарии переходов, они образуют её Feature Navigation. Если таких переходов нет, отдельный navigation-слой не вводится. Feature Navigation не должна становиться глобальным маршрутизатором приложения.

## Presentation Navigation

Presentation Navigation отвечает за временные presentation-flow поверх основных экранов.

Она может открывать временный сценарий или передавать навигационное намерение в основной navigation-контракт, но не должна подменять root navigation или владеть структурой приложения.

## Navigation State

Navigation State хранит выбранный корневой раздел, локальный feature-путь или одноразовое навигационное намерение.

Navigation State отделён от domain-состояния, Runtime Metadata и Track Identity. Он описывает, куда перейти, а не выполняет системную операцию.

# Поток данных

Типовой поток навигационного действия:

```text
Пользовательское действие
↓
Screen отправляет Action
↓
Слой обработки действия определяет навигационное намерение
↓
Navigation обновляет navigation state или presentation-flow
↓
Root Screen или feature-flow получает новое состояние
↓
Пользователь видит новый экран, detail-flow или presentation-flow
```

Для межсценарного перехода поток остаётся тем же: источник создаёт навигационное намерение, общий navigation-контракт переводит пользователя к нужной root-точке, а целевая feature-подсистема применяет это намерение внутри собственного flow.

# Контракты

- Navigation определяет переходы между сценариями, но не содержит бизнес-логику.
- Navigation определяет владельца перехода и целевой сценарий, но не визуальный способ показа перехода.
- Выбор корневого раздела является частью общего Navigation-контракта и отделён от маршрутов внутри feature.
- Root Screen является владельцем собственного navigation-flow.
- Feature-подсистема управляет только своей внутренней навигацией.
- Если feature-подсистема содержит собственные сценарии переходов, они образуют её Feature Navigation.
- Presentation Navigation не изменяет структуру основной навигации.
- Presentation-flow может запросить переход через общий navigation-контракт, но не становится владельцем root navigation.
- Navigation State не подменяет domain-состояние, Runtime Metadata или Track Identity.
- UI передаёт пользовательское намерение наружу и не становится владельцем navigation-flow.
- Если навигационное намерение относится к физическому треку, оно должно опираться на Track Identity.

# Инварианты

- Новая feature-подсистема не создаёт независимую глобальную систему навигации.
- Навигация отделена от бизнес-логики, файловых операций, storage и внешних интеграций.
- Экран не становится владельцем чужого navigation-flow.
- Модальные или временные presentation-flow не подменяют root-навигацию.
- Одноразовые навигационные намерения не становятся постоянным доменным состоянием.
- Дочерний UI-компонент не открывает чужой сценарий напрямую, если для этого есть root, feature или presentation-контракт.
- Если feature не содержит собственных сценариев переходов, для неё не вводится лишний navigation-слой.

# Зависимости

Navigation используется документами и будущими каноничными описаниями следующих подсистем:

- Overview — место Navigation в общей архитектуре BurningTrack.
- Root Screens — владение screen-flow на уровне корневых экранов.
- Sheets — правила временных presentation-flow.
- Library — feature-навигация фонотеки.
- Player — presentation-действия и переходы из player-flow.
- Tracklists — master-detail и presentation-действия треклистов.
- Track Identity — идентичность целей навигации, связанных с физическим треком.
