# Назначение

Sheets — это архитектурный контракт временных presentation-flow в BurningTrack.

Они нужны для пользовательских сценариев, которые открываются поверх текущего root screen-flow или feature-flow, имеют ограниченный жизненный цикл и после завершения возвращают управление владельцу исходного сценария.

Sheets не являются Root Navigation, не являются самостоятельной feature-подсистемой и не определяют владение постоянными данными. Их архитектурная роль — изолировать временный сценарий от основной навигации и feature-состояния.

# Проблема

Если временные пользовательские сценарии становятся частью Root Navigation или Feature Flow, границы ответственности начинают размываться:

- root-навигация начинает хранить краткоживущие состояния;
- feature-flow начинает отвечать за временные presentation-сценарии, которые не являются частью его основной структуры;
- закрытие временного сценария перестаёт быть явной границей lifecycle;
- результат, отмена и переход к следующему временному сценарию начинают обрабатываться локально и непредсказуемо;
- UI может начать выполнять системные операции или менять навигацию напрямую;
- становится сложнее понять, кто владеет состоянием, действием и результатом сценария.

Отдельный контракт Sheets нужен, чтобы временные сценарии были ограничены presentation boundary и не подменяли root navigation, feature-flow, Runtime Metadata или Track Identity.

# Архитектурная схема

Место Sheets в архитектуре BurningTrack:

```text
Root Screen
↓
Feature Flow
↓
Presentation Boundary
↓
Sheet Flow
↓
Sheet State / Sheet Action
↓
Результат или отмена
↓
Feature Flow
```

Если временный сценарий должен открыть другой пользовательский сценарий или перейти к другому разделу, он делает это через существующий navigation-контракт:

```text
Sheet Flow
↓
Навигационное намерение
↓
Navigation
↓
Root Screen / Feature Flow
```

Схема описывает архитектурные роли, а не визуальный способ показа временного сценария.

# Основные компоненты

## Sheet Flow

Sheet Flow — ограниченный пользовательский сценарий, который существует поверх текущего root screen-flow или feature-flow.

Он управляет только временным сценарием: принимает пользовательские действия, хранит transient-состояние, формирует результат или отменяет сценарий. Sheet Flow не становится владельцем исходного Feature Flow.

## Sheet State

Sheet State описывает готовое состояние временного сценария.

Это состояние ограничено жизненным циклом Sheet Flow. Оно может содержать черновики, выбор пользователя, режим сценария или данные для подтверждения, но не является постоянным storage, Runtime Metadata или domain-состоянием.

## Sheet Action

Sheet Action описывает пользовательское намерение внутри временного сценария.

Action не выполняет работу самостоятельно. Он передаёт смысл действия в слой обработки временного сценария, который либо обновляет Sheet State, либо делегирует системную операцию нижележащему контракту.

# Поток данных

Типовой жизненный цикл временного сценария:

```text
Пользовательское действие
↓
Feature Flow
↓
Открытие Sheet через Presentation Boundary
↓
Sheet Flow получает входной контекст
↓
Sheet State отображается пользователю
↓
Пользователь отправляет Sheet Action
↓
Sheet Flow обновляет временное состояние или формирует результат
↓
Результат применяется через соответствующий feature, domain или navigation-контракт
↓
Sheet Flow завершается
↓
Управление возвращается владельцу Feature Flow
```

Результатом временного сценария может быть подтверждение, отмена, запрос системного действия или переход к следующему временному сценарию. Результат не должен превращать Sheet в владельца feature-данных или root-навигации.

Если действие меняет данные трека, обновление отображения должно вернуться через соответствующий runtime-контракт. Если действие относится к физическому треку, оно должно использовать Track Identity. Если действие требует перехода, оно должно пройти через Navigation.

# Контракты

- Sheet является временным presentation-flow поверх текущего root screen-flow или feature-flow.
- Sheet не становится частью Root Navigation.
- Sheet не становится владельцем Feature Flow.
- Sheet имеет собственные State и Action или эквивалентный контракт временного состояния и пользовательских действий.
- Presentation Boundary отвечает за lifecycle временного сценария и не содержит бизнес-логику.
- После завершения Sheet управление возвращается владельцу Feature Flow или передаётся следующему архитектурному контракту.
- Sheet может инициировать изменение данных, но не становится владельцем этих данных.
- Системные операции выполняются через существующие feature, domain, command или runtime-контракты.
- Переход из Sheet в другой сценарий выполняется через Navigation, а не через прямое владение чужим flow.
- Sheet State является transient-состоянием и не подменяет Runtime Metadata, Track Identity или persistent storage.

# Инварианты

- Временный сценарий не изменяет структуру Navigation.
- Sheet не становится самостоятельной feature-подсистемой.
- Жизненный цикл Sheet ограничен текущим пользовательским сценарием.
- Закрытие Sheet завершает его Flow или передаёт управление следующему явно открытому временному сценарию.
- Transient-состояние Sheet можно отбросить или пересобрать без изменения архитектурного контракта.
- Sheet не читает файлы, storage или внешние интеграции напрямую, если для этого существует нижележащий контракт.
- Дочерний UI временного сценария не становится владельцем feature-данных, navigation-flow или runtime-обновлений.

# Зависимости

Sheets используются документами и будущими каноничными описаниями следующих подсистем:

- Overview — место Sheets в общей архитектуре BurningTrack.
- Root Screens — граница root screen-flow, поверх которой открываются временные сценарии.
- Navigation — правила переходов из временного сценария в другой flow.
- Runtime Metadata — обновление отображаемых данных после действий во временном сценарии.
- Track Identity — идентичность физического трека внутри временных сценариев.
- Library — временные presentation-flow фонотеки.
- Player — временные presentation-flow playback-сценариев.
- Tracklists — временные presentation-flow сценариев треклистов.
- Batch Operations — временные presentation-flow массовых действий.
- Track Edit Session — контракт редактирования внутри временного сценария.
