# Назначение

Development Rules определяет единые правила разработки BurningTrack.

Документ служит соглашением для всех изменений проекта: новая работа должна сохранять архитектурную целостность, уважать существующие границы ответственности и опираться на уже принятые проектные контракты.

# Общие принципы

- Соблюдать существующую архитектуру проекта.
- Не нарушать границы ответственности подсистем.
- Сначала использовать существующие контракты, а не создавать новые.
- Не дублировать существующую функциональность.
- Временные решения допустимы только как явно отмеченное исключение.
- Любое изменение должно улучшать или сохранять архитектурную целостность проекта.
- Если архитектурный вопрос уже раскрыт в каноничном документе, использовать ссылку на него вместо повторного описания.

# Правила разработки

- Новая функциональность сначала ищет существующего владельца ответственности.
- Новый контракт появляется только при отсутствии подходящего существующего контракта.
- Изменение не должно нарушать публичный контракт существующей подсистемы.
- Рефакторинг не должен менять пользовательское поведение без отдельной задачи.
- Изменение должно быть завершённым и не оставлять архитектуру в промежуточном состоянии.
- Предпочтение отдаётся устранению причины проблемы, а не её симптомов.
- Если изменение требует временного решения, оно должно быть явно обозначено и ограничено по области применения.
- Если изменение меняет архитектурный контракт, соответствующий каноничный документ должен быть обновлён.

# Ограничения

- Этот документ не определяет архитектуру приложения.
- Этот документ не заменяет архитектурные документы.
- Этот документ не описывает правила оформления документации.
- Этот документ не описывает стиль кода.
- Этот документ не определяет workflow Git.
- Этот документ не описывает устройство отдельных подсистем.
