# Правила архитектурной документации

## Назначение

Этот документ определяет единый стандарт архитектурной документации проекта BurningTrack.

Главный принцип:

Документация описывает устойчивую архитектуру проекта, а не текущую реализацию.

Если внутренний рефакторинг не изменил архитектурный контракт, документация не должна требовать изменений.

---

## Для кого предназначена документация

Документация должна помогать разработчику:

- быстро понять устройство подсистемы;
- понять её место в общей архитектуре проекта;
- безопасно развивать систему без нарушения архитектурных контрактов;
- быстро найти документ, отвечающий на конкретный архитектурный вопрос.

Документация не предназначена для изучения деталей реализации Swift-кода.

---

## Общие правила

- Один документ отвечает на один архитектурный вопрос.
- Один документ описывает одну подсистему.
- Для каждого архитектурного вопроса существует один каноничный документ.
- Остальные документы должны ссылаться на него, а не повторять его содержание.
- Документы не дублируют друг друга.
- Если тема подробно раскрыта в другом документе, используется ссылка, а не повторение текста.
- Документ описывает архитектурные принципы и контракты, а не детали реализации.
- История архитектурных решений хранится только в ADR.
- Детали реализации допускаются только тогда, когда без них невозможно понять архитектурный контракт или существующее ограничение.

---

## Что должно быть в документации

Документ описывает только устойчивые элементы архитектуры:

- назначение подсистемы;
- её место в общей архитектуре;
- зоны и границы ответственности;
- поток данных;
- взаимодействие с другими подсистемами;
- архитектурные контракты;
- инварианты;
- ограничения;
- связанные документы.

---

## Что не должно попадать в документацию

Не описываются детали реализации, которые могут измениться без изменения архитектуры.

Например:

- property wrapper (`@State`, `@Published`, `@Observable` и т.п.);
- внутренние методы;
- детали подписок;
- временные вспомогательные методы;
- внутренние вызовы (`refresh()`, `refreshState()` и т.п.);
- использование `Task`, `Combine`, `Observation`;
- детали SwiftUI;
- модификаторы представлений;
- порядок вызова внутренних функций;
- большие фрагменты кода;
- реализация конкретных алгоритмов, если они не являются архитектурным контрактом.

Если пример кода необходим для понимания архитектурного контракта, он должен быть минимальным и иллюстрировать только сам контракт, а не реализацию.

---

## Уровень абстракции

Документация должна отвечать на вопрос:

Как устроена подсистема?

Документация не должна отвечать на вопрос:

Как именно написан код?

Если реализацию можно полностью переписать без изменения архитектурного контракта, документ должен остаться актуальным.

---

## Терминология

### Общепринятые архитектурные роли

Для общепринятых архитектурных ролей используются оригинальные англоязычные названия.

Например:

- Screen;
- ViewModel;
- State;
- Action.

Эти термины являются частью общепринятой архитектурной терминологии и должны совпадать с терминологией, используемой в коде проекта.

Их не рекомендуется переводить на русский язык.

### Роли, зависящие от реализации

Компоненты, являющиеся особенностью текущей реализации проекта, не должны описываться через конкретные названия типов.

Вместо этого следует использовать описание их архитектурной ответственности.

Например:

- слой формирования состояния;
- слой обработки пользовательских действий;
- слой сборки зависимостей.

При необходимости допускается один раз указать, какие типы текущей реализации выполняют эти роли.

Главным остаётся описание архитектурной ответственности, а не название конкретного класса.

---

## Форматы документов

### Architecture document

Используется для документов, описывающих архитектурные подсистемы.

Например:

- Root Screens;
- Navigation;
- Runtime Metadata;
- Track Identity;
- Sheets;
- Errors and Toasts;
- Mini Player.

Структура:

- Назначение;
- Проблема;
- Архитектурная схема;
- Основные компоненты;
- Поток данных;
- Контракты;
- Инварианты;
- Зависимости;
- Связанные документы.

---

### Feature document

Используется для документов, описывающих функциональные возможности приложения.

Например:

- Library;
- Player;
- Tracklists;
- Batch Operations;
- Track Edit Session;
- Export.

Структура:

- Назначение;
- Пользовательский сценарий;
- Основной поток;
- Участвующие подсистемы;
- Контракты;
- Ограничения;
- Связанные документы.

---

### Integration document

Используется для описания внешних интеграций.

Например:

- TagLib;
- будущие интеграции Rekordbox;
- облачные сервисы;
- внешние SDK.

Структура:

- Назначение;
- Используемая технология;
- Как подключена;
- Какие подсистемы используют;
- Ограничения;
- Правила обновления;
- Связанные документы.

---

## Правила написания

Перед созданием нового документа необходимо определить:

1. Какой архитектурный вопрос он должен закрыть.
2. Нет ли уже документа, отвечающего на этот вопрос.
3. Не дублирует ли новый документ существующую информацию.
4. Можно ли вместо повторения сослаться на другой документ.

Если часть информации относится к другой подсистеме, она не описывается подробно, а сопровождается ссылкой на соответствующий документ.

---

## Чек-лист перед завершением документа

Перед сохранением документа необходимо ответить на следующие вопросы:

- Документ отвечает только на один архитектурный вопрос?
- Есть ли в нём дублирование с другими документами?
- Попали ли в него детали реализации, не влияющие на архитектуру?
- Понятны ли архитектурные контракты и ограничения?
- Понятны ли границы ответственности подсистемы?
- Останется ли документ актуальным после внутреннего рефакторинга?
- Поймёт ли новый разработчик устройство подсистемы, не изучая реализацию?
