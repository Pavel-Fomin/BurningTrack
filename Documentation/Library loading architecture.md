# Архитектура загрузки треков фонотеки

## Назначение

Архитектура загрузки фонотеки должна обеспечивать почти мгновенное открытие папок с сотнями и тысячами треков.

Первый экран списка треков не должен ждать:

- файловую систему;
- восстановление bookmark;
- синхронизацию фонотеки;
- расчёт badges;
- проверку доступности файлов;
- чтение тегов, длительности и обложек.

## Основной принцип

Первичная загрузка списка и тяжёлая догрузка деталей разделены.

```text
Вход в папку
→ быстрый список из TrackRegistry
→ первый показ UI
→ фоновая догрузка деталей
```

## Быстрый первый показ

Первичный список строится через:

- `LibraryTracksViewModel.loadInitialTracks()`;
- `FastLibraryTracksProvider`;
- `TrackRegistry.tracks(inFolder:)`.

В первичном показе запрещено:

- вызывать `MusicLibraryManager.syncFolderIfNeeded(...)`;
- вызывать `BookmarkResolver.url(forTrack:)`;
- читать `url.resourceValues(...)`;
- считать badges;
- проверять доступность файлов;
- читать теги и обложки.

`FastLibraryTracksProvider` должен работать только с данными, уже сохранёнными в `TrackRegistry`.

## TrackRegistry как источник первичного списка

`TrackRegistry` хранит минимальные данные, достаточные для быстрого отображения списка:

- `track id`;
- `fileName`;
- `relativePath`;
- `folderId`;
- `rootFolderId`;
- `importedAt`;
- `updatedAt`;
- `fileDate`.

Смысл дат:

- `importedAt` — дата первого появления трека в фонотеке;
- `updatedAt` — дата обновления записи в реестре;
- `fileDate` — сохранённая дата файла для сортировки и группировки.

`fileDate` читается во время sync и сохраняется в `TrackRegistry`.
При открытии папки `fileDate` не читается из файловой системы.

## Секции и сортировка

Секции фонотеки строятся по сохранённой `fileDate`.

Это позволяет отображать реальные даты файлов без обращения к файловой системе при входе в папку.

Список не должен перестраиваться только из-за фоновой проверки доступности. Фоновая проверка может обновлять признаки строк, но не должна менять порядок треков и секций.

## Фоновая догрузка

После первого показа `LibraryTracksViewModel.loadDetailsInBackground()` выполняет тяжёлые операции:

- `syncFolderIfNeeded(...)`;
- пересчёт badges;
- фоновую проверку доступности файлов.

При фоновой догрузке:

- порядок треков не должен меняться;
- секции не должны пересобираться;
- UI может обновлять только дополнительные признаки строки, например `isAvailable` и badges.

## Доступность файлов

При первичном показе треки считаются доступными.

Реальная доступность проверяется после первого показа. Проверка может использовать `BookmarkResolver.url(forTrack:)`, потому что она уже находится вне первичного пути загрузки.

Результат проверки записывается в `LibraryTrack.isAvailable`.

`TrackRowView` уже умеет визуально приглушать недоступные треки и блокировать тап по ним.

## Badges

Badges догружаются после первого показа списка.

Текущая реализация `DefaultTrackListBadgeProvider` остаётся отдельной зоной для будущего рефакторинга.

Расчёт badges нельзя возвращать в первичный путь загрузки.

## Запреты

В быстром пути загрузки запрещено:

- возвращать `BookmarkResolver` в `FastLibraryTracksProvider`;
- читать `resourceValues` при открытии папки;
- запускать sync до первого присваивания `trackSections`;
- считать badges до первого отображения списка;
- использовать `updatedAt` для группировки треков по дате файла.

## Связь с будущей БД

Текущий JSON-контракт `TrackRegistry` подготовлен к будущей миграции в БД.

Поля `fileDate`, `importedAt` и `updatedAt` должны переехать в таблицу треков без изменения смысла:

- `fileDate` — дата файла для сортировки и группировки;
- `importedAt` — дата первого появления трека в фонотеке;
- `updatedAt` — дата обновления записи.
